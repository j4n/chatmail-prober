# cmping-exporter: Smokeping-style Prometheus Exporter for Chatmail Relay Interop

## Context

GitHub issue [scalable-resilient-chat#5](https://github.com/chatmail/scalable-resilient-chat/issues/5) requires running interop tests "at least daily" across chatmail relays. The test tool (`cmping`) already exists. The missing piece is **automated, periodic probing with Prometheus-exportable metrics** — modeled after [smokeping_prober](https://github.com/SuperQ/smokeping_prober).

This will be a **separate project/repo**.

## Parallelism via per-pair accounts directories

cmping's `deltachat-rpc-server` holds an exclusive lock on its accounts database. To parallelize, each relay pair gets its **own accounts directory** (and thus its own DB/RPC server):

```
~/.cache/cmping-exporter/
├── nine.testrun.org--nine.testrun.org/    # self-loop
├── nine.testrun.org--mehl.cloud/          # pair A→B
├── mehl.cloud--nine.testrun.org/          # pair B→A
└── ...
```

This means each relay accumulates multiple accounts (one per pair it participates in). For 3 relays that's 3 accounts per relay — trivial. For 30 relays it'd be 30 per relay, still fine for chatmail.

Starting with 3 relays (nine.testrun.org, mehl.cloud, tarpit.fun) = 9 pairs:
- `-c 10` at 1.1s interval + ~5s overhead = ~16s per pair
- 9 pairs, 5 workers → 2 batches × ~16s = **~32s** per round
- Scales: 30 relays (900 pairs), 20 workers → 45 batches × ~16s = **~12 min**

## Architecture

A long-running Python process that:

1. **Parallel probe loop** — every 15 minutes, probes all relay pairs concurrently via thread pool (10 pings each for smokeping-style distribution)
2. **Per-pair accounts dirs** — each pair gets its own DB, avoiding the deltachat-rpc-server lock
3. **HTTP `/metrics` endpoint** and/or **textfile output** — dual mode
4. **Config file** — plain text list of relay domains (start with 3: nine.testrun.org, mehl.cloud, tarpit.fun)
5. **Persistent account cache** — reuses chatmail accounts across runs within each pair's directory

```
┌──────────────────┐  thread pool   ┌─────────┐
│ probe scheduler  │═══ worker 1 ══▶│ A → B   │  (own accounts dir)
│                  │═══ worker 2 ══▶│ B → A   │  (own accounts dir)
│ ThreadPoolExec   │═══ worker 3 ══▶│ A → C   │  (own accounts dir)
│ (N workers)      │═══ worker 4 ══▶│ C → A   │  (own accounts dir)
│                  │═══ ...      ══▶│ ...     │
└──────┬───────────┘                └─────────┘
       │ updates metrics (thread-safe)
       ▼
┌──────────────────┐   :9740/metrics     ┌─────────────┐
│ prometheus_client│◀── scrape ──────────│ Prometheus   │
│ HTTP server      │                     └─────────────┘
│                  │
│ (+ optional      │──▶ /var/lib/prometheus/node-exporter/cmping.prom
│  textfile write) │
└──────────────────┘
```

### Dual output mode

Supports both output methods — use whichever fits the deployment:

| Flag | Mode | When to use |
|------|------|-------------|
| `--port 9740` (default) | **HTTP exporter** | Standalone deployment, Prometheus scrapes directly |
| `--textfile /var/lib/prometheus/node-exporter/cmping.prom` | **Textfile** | node_exporter already running, write `.prom` file to its textfile collector dir |
| Both | **Both** | HTTP endpoint + textfile simultaneously |

Textfile mode uses atomic write (write to `.tmp`, then rename) — same pattern as the existing `fsreport.py` in chatmaild.

## Prerequisite: cmping upstream PR ("make importable")

A dedicated PR to [chatmail/cmping](https://github.com/chatmail/cmping) making it safe to use as a library. The current code has several import-hostile patterns:

### Changes needed

1. **Replace `sys.exit(1)` with exceptions** — currently in `setup_accounts()` (lines 301, 313) and `wait_profiles_online()` (line 534). These kill the entire process. Define a `CmpingError` exception class and raise it instead:
   ```python
   class CmpingError(Exception):
       pass

   # In setup_accounts(), replace:
   #   sys.exit(1)
   # with:
   #   raise CmpingError(f"Failed to setup sender profile on {args.relay1}: {e}")
   ```
   The CLI `main()` catches `CmpingError` and calls `sys.exit(1)` — behavior unchanged for CLI users.

2. **`Pinger.results` list** — collect `(seq, ms_duration, receiver_idx)` tuples so consumers get structured RTT data:
   ```python
   # In Pinger.__init__:
   self.results = []

   # In perform_ping(), in the receive loop, alongside the existing tracking:
   pinger.results.append((seq, ms_duration, receiver_idx))
   ```

3. **`perform_ping()` timing data** — the function already computes `account_setup_time`, `group_join_time`, `message_time` but only prints them. Return them on the `Pinger` object:
   ```python
   pinger.account_setup_time = account_setup_time
   pinger.group_join_time = group_join_time
   pinger.message_time = message_time
   ```

4. **Configurable `accounts_dir`** — currently hardcoded to `xdg_cache_home()/cmping`. Add an `accounts_dir` parameter to `perform_ping()` so the exporter can isolate each pair:
   ```python
   def perform_ping(args, accounts_dir=None):
       base_accounts_dir = accounts_dir or xdg_cache_home().joinpath("cmping")
   ```
   This is the key enabler for parallel probing — each pair uses a separate DB directory.

5. **Quiet mode** (nice-to-have) — add `quiet=False` parameter or respect `verbose=-1` to suppress all print output. For now the exporter can just redirect stdout.

## cmping internals (from source analysis)

cmping is a single module (`cmping.py`). Key pieces for library use:

| Function / Class | What it gives us |
|---|---|
| `perform_ping(args)` | Top-level entry; returns `Pinger` with `.sent`, `.received`, `.loss` |
| `Pinger.receive()` | Generator yielding `(seq, ms_duration, size, receiver_idx)` — **individual RTTs** |
| `AccountMaker.get_relay_account(domain)` | **Reuses existing accounts** from `~/.cache/cmping/{relay}/` |
| `args` | argparse `Namespace` — construct manually for library use |

**Account reuse is built in:** cmping stores accounts under `xdg_cache_home()/cmping/{relay}/`. Accounts persist across runs. No throwaway churn.

## Metrics

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `cmping_requests_total` | Counter | `source`, `destination` | Pings sent |
| `cmping_response_duration_seconds` | Histogram | `source`, `destination` | Round-trip time per message |
| `cmping_responses_total` | Counter | `source`, `destination` | Successful responses |
| `cmping_send_errors_total` | Counter | `source`, `destination` | Failed probes (timeout, crash) |
| `cmping_probe_success` | Gauge | `source`, `destination` | 1 if last probe round succeeded |
| `cmping_account_setup_seconds` | Gauge | `source`, `destination` | Account setup time (high on first run) |

Histogram buckets for email RTT: `[0.1, 0.25, 0.5, 1, 2.5, 5, 10, 30, 60]` seconds.

## Relay pair testing

All ordered pairs **including self-loops** (A→A). Self-probes serve as baseline — if A→A fails, the relay itself is down.

Sequential execution. Metrics update **after each pair completes**, so Prometheus sees partial results during a round.

## Implementation sketch

### prober.py

```python
import argparse, contextlib, io, time
from dataclasses import dataclass, field
from cmping import perform_ping

@dataclass
class ProbeResult:
    source: str
    destination: str
    sent: int = 0
    received: int = 0
    loss: float = 100.0
    rtts_ms: list[float] = field(default_factory=list)
    setup_time: float = 0
    error: str | None = None

def run_probe(source: str, dest: str, count: int = 10,
              cache_dir: str = "~/.cache/cmping-exporter") -> ProbeResult:
    # Per-pair accounts dir avoids DB lock between concurrent probes
    pair_dir = Path(cache_dir).expanduser() / f"{source}--{dest}"
    args = argparse.Namespace(
        relay1=source, relay2=dest,
        count=count, interval=1.1, verbose=0,
        numrecipients=1, reset=False,
    )
    try:
        with contextlib.redirect_stdout(io.StringIO()):
            pinger = perform_ping(args, accounts_dir=pair_dir)  # upstream PR
        return ProbeResult(
            source=source, destination=dest,
            sent=pinger.sent, received=pinger.received,
            loss=pinger.loss,
            rtts_ms=[rtt for _, rtt, _ in pinger.results],
            setup_time=pinger.account_setup_time,
        )
    except (SystemExit, Exception) as e:
        return ProbeResult(source, dest, error=str(e))
```

### __main__.py

```python
def main():
    args = parse_args()
    relays = read_relay_list(args.relays)

    if args.port:
        start_http_server(args.port)

    while True:
        round_start = time.time()
        pairs = [(s, d) for s in relays for d in relays]

        with ThreadPoolExecutor(max_workers=args.workers) as pool:
            futures = {
                pool.submit(run_probe, src, dst, args.count, args.cache_dir): (src, dst)
                for src, dst in pairs
            }
            for future in as_completed(futures):
                result = future.result()
                update_metrics(result)
                log_result(result)

        if args.textfile:
            write_textfile(args.textfile)  # atomic write .prom

        elapsed = time.time() - round_start
        remaining = max(0, args.interval - elapsed)
        if remaining == 0:
            log.warning(f"Probe round took {elapsed:.0f}s, exceeds interval {args.interval}s")
        time.sleep(remaining)
```

## Future considerations

- **Scaling to all 30 public relays**: 900 pairs with 20 workers at `-c 10` ≈ 12 min. Feasible but resource-heavy (20 concurrent RPC servers + IMAP connections). May want to limit `--workers` or reduce `-c`.
- **Alertmanager integration**: Route `cmping_probe_success == 0` alerts for relay outages.
- **Dynamic relay list**: Scrape https://chatmail.at/relays to auto-discover public relays instead of a static config file.
